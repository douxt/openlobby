# AI 辅助编程开发指南（v4.0）

> **信息来源**：Matt Pocock 在 AI Engineer 大会上的深度工作坊《Full Walkthrough: Workflow for AI Coding》。Matt 是 TypeScript 专家、Total TypeScript 创始人，过去半年专注于探索如何将软件工程基本功与 AI 工具高效结合。本手册基于该工作坊的核心方法，结合社区最佳实践综合整理而成。
>
> **v4.0 修订**：新增 `/zoom-out`（全局视角）、`/diagnose`（系统化排查）、重试限制、Git 工作流规范、新手入门路径。
>
> 来源：[AI开发指南.md](https://github.com/anthropics/claude-code) — 基于真实项目 200+ 次 AI 辅助开发迭代验证。

## 第一部分：核心理念

1. **软件工程基础原则对 AI 同样有效**：小任务、快速反馈、测试驱动、模块化设计。
2. **AI 不是"规格到代码"的编译器**，而是需要对齐和引导的协作伙伴。
3. **人类负责方向与品味，AI 负责实现与测试**。
4. **始终保持 LLM 在"智能区"（Smart Zone）**：约 **100K token** 以内，超过即进入"愚蠢区"。
5. **阶段切换时：清除上下文，信息通过文档传递**。这是确保可复现、可审计、无污染的核心原则。

## 第二部分：阶段间协作规范

| 做法 | 效果 |
|------|------|
| **清除上下文** | 每个阶段都在"干净的智能区"开始，不受历史噪声干扰 |
| **信息固定到文档** | PRD、issue 文件等成为单一事实来源 |

**实操规范**：
- 每个阶段结束时：产物写入磁盘 → 执行 `/clear`（或新建对话）
- 每个阶段开始时：明确告诉 AI 读取哪些文档，不依赖对话记忆

**例外**：探索/原型阶段可暂时使用压缩；正式开发阶段坚决清除。

## 第三部分：完整工作流程（5 个阶段）

### 阶段 0：环境准备

| 步骤 | 操作 |
|------|------|
| 0.1 | 安装技能包：`npx skills@latest add mattpocock/skills` |
| 0.2 | 选择 Issue 追踪器（本地 markdown / GitHub Issues） |
| 0.3 | 初始化项目目录，生成 `CLAUDE.md` |
| 0.4 | 配置领域术语 `CONTEXT.md` |
| 0.5 | 启用 Token 用量监控，接近 80K 时清除上下文 |
| 0.6 | 确保快速反馈：单元测试 < 10s，类型检查 < 5s |

**核心技能文件**（`.claude/` 目录下）：
- `/grill-me`：全方位拷问设计
- `/zoom-out`：全局视角审视代码库，生成架构概览
- `/diagnose`：多步骤系统化排查流程
- `/to-prd`：将对话转成 PRD
- `/to-issues`：将 PRD 拆成垂直切片 issue
- `/tdd`：强制执行红→绿→重构循环
- `/improve-codebase-architecture`：寻找架构改进点

### 阶段 1：对齐（Grill Me）

| 步骤 | 操作 |
|------|------|
| 1.1 | 清除上下文，开始新会话 |
| 1.2 | 调用 `/grill-me <需求文件>` |
| 1.3 | AI 派探索子代理扫描代码库，逐一提问 |
| 1.4 | 人类回答每个问题（可接受推荐答案） |
| 1.5 | 持续直到关键决策被覆盖（20–80 个问题） |

**阶段结束**：关键结论记录到设计笔记（可选），**清除上下文**。

### 阶段 2：PRD（快速固化）

| 步骤 | 操作 |
|------|------|
| 2.1 | 新建会话，读取上阶段产出 |
| 2.2 | 调用 `/to-prd`（附带 PRD 质量宪法） |
| 2.3 | AI 生成 PRD，人类快速浏览（≤2 分钟），重点确认"范围外" |
| 2.4 | 将 PRD 提交到仓库，**清除上下文** |

### 阶段 3：拆解为可并行任务

| 步骤 | 操作 |
|------|------|
| 3.1 | 新建会话，读取 PRD |
| 3.2 | 调用 `/to-issues`（附带 Issue 质量宪法），强制要求垂直切片 |
| 3.3 | 检查阻塞关系，确认 DAG |
| 3.4 | 产出 `issues/` 目录，提交仓库，**清除上下文** |

### 阶段 4：AFK 实现 — AI 自主编码

#### 4.0 TDD 前置准备（必需）
- [ ] 确保测试框架可运行
- [ ] 运行 `/tdd` 技能
- [ ] 用最小 issue 手动验证一次（AI 先写失败测试）

#### 4.1 系统化排查流程（/diagnose）

当 AI 遇到 bug 时，**不允许直接猜答案**，必须调用 `/diagnose`：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1. 复现 | 描述问题现象，提供复现步骤 | 例如："运行测试时第 42 行报错：预期 10，实际 0" |
| 2. 缩小范围 | 隔离相关代码模块，添加临时日志或断点 | "在 X 函数中添加日志观察变量 Y 的值" |
| 3. 提出假设 | 基于证据提出至少一个明确假设 | "假设是数据库查询未返回最近完成的记录" |
| 4. 验证假设 | 通过修改代码或运行诊断命令确认 | 手动执行查询，或临时修改返回值 |
| 5. 修复与回归测试 | 实施修复，运行完整测试套件 | 确保原问题消失且未引入新问题 |

> **禁止**：AI 跳过上述步骤直接输出"修复代码"。

#### 4.2 反馈闭环与重试限制

- **最大重试次数**：**2-3 次**（针对同一问题）
- **超出限制后**：AI 输出 `[UNRESOLVED]` 报告，建议人类介入

#### 4.3 运行 AFK 循环

```bash
bash ralph-once.sh   # 首次必须手动观察
```

### 阶段 5：QA & 代码审查

| 步骤 | 操作 |
|------|------|
| 5.1 | 手动验证功能，尤其前端视觉 |
| 5.2 | **审查测试**：确认测试有意义（非假测试） |
| 5.3 | 检查安全风险（SQL 注入、权限） |
| 5.4 | 发现问题 → 创建新 issue（含复现步骤），不直接修改 AI 代码 |
| 5.5 | 若整体满意，合并代码 |

## 第四部分：Git 工作流规范

| 分支 | AI 权限 | 说明 |
|------|---------|------|
| `main` / `master` | **禁止直接推送** | 人类保留唯一写入权限 |
| 特性分支 | AI 可创建和推送 | 命名格式：`ai/<issue-id>-<short-desc>` |

- **所有 AI 代码变更必须通过 PR 提交**
- PR 必须经过至少一名人类审查后方可合并
- 若 AI 尝试直接 push 到 main，CI 应拒绝并发出警告

## 第五部分：质量保证体系

### 5.1 过程质量

| 手段 | 解决的问题 |
|------|------------|
| Grill Me | 假设对齐偏差 |
| 垂直切片 + 曳光弹 | 早期可验证 |
| TDD（红-绿-重构） | 测试作弊 |
| 快速反馈循环（< 10s） | AI 盲写 |
| 系统化排查（/diagnose） | AI 猜谜式调试 |
| 重试限制（≤3 次） | 无限循环浪费 |

### 5.2 架构质量 — 深模块 vs 浅模块

AI 天生倾向浅模块，人类必须定期纠偏。每月运行 `/improve-codebase-architecture`。

### 5.3 文档保鲜

完成后关闭 issue，不长期保留过时 PRD。

## 第六部分：新手快速入门

首次接触建议在 **2-4 小时**内完成端到端试运行：

1. **选择小型独立项目**（30min）— 个人工具脚本或独立模块，确保有测试框架
2. **运行 Grill Me**（30-60min）— `/grill-me <你的需求>`，回答 AI 的所有问题
3. **生成 PRD 并拆解 Issue**（20min）— `/to-prd` → `/to-issues`
4. **TDD 验证**（15min）— 选择最简单的 1 个 issue，观察 AI 是否先写失败测试
5. **AFK 运行单个 issue**（10min）— 审查产出：测试是否合理？代码是否可读？
6. **反思与调整**（15min）— 记录问题，调整 prompt 或拆分粒度

## 第七部分：常见错误流程对比

| 错误做法 | 正确做法 |
|----------|----------|
| 阶段切换时使用压缩 | 阶段切换时**清除上下文** |
| 依赖对话记忆传递信息 | **通过文档传递信息** |
| to-issues 后立即跑 AFK | 先做 TDD 前置准备 |
| AI 先写实现再补测试 | **强制 TDD：先写失败测试** |
| AI 遇到 bug 直接猜答案 | 调用 `/diagnose` 走系统化排查 |
| AI 尝试无限次自我修复 | **限制重试 ≤ 3 次**，超出转人类 Issue |
| AI 直接 push 到 main | **所有变更通过 PR**，人类审查合并 |
| 直接修改 AI 代码 | 创建新 issue 让 AI 修复 |

## 第九部分：术语表

| 术语 | 含义 |
|------|------|
| **智能区** | LLM 保持高质量推理的上下文范围（~100K token） |
| **垂直切片** | 端到端跨全栈的小功能 |
| **深模块** | 小接口、大实现，易于测试和委托 |
| **AFK** | Away From Keyboard，人类离开 AI 自主执行 |
| **Ralph 循环** | Matt 命名的 AFK 自动编码循环 |
| **/diagnose** | 系统化排查技能，强制多步骤诊断流程 |
| **/zoom-out** | 全局视角技能，生成架构概览 |
