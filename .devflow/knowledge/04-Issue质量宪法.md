# Issue 质量宪法（14 项）

> 每次生成、修改 issue 时，必须逐条对照以下 14 项。缺任一项的 issue 不得置为 `ready`。

## 前置指导（写 issue 时逐条对照）

| # | 检查项 | 规则 | 违规后果 |
|---|--------|------|----------|
| 1 | **工时优先 ≤1d** | estimate 字段 ≤1d。除非拆分会留无法独立验证的半成品，可放宽到 2d | AFK 上下文超限 |
| 2 | **type 正确** | AFK 不含人工步骤，HITL 含人工步骤 | Agent 无法完成 |
| 3 | **AC 可测量** | 每条 AC 可量化或可自动化断言 | Agent 自判"完成"但实际未完成 |
| 4 | **目录已指定** | AC 或正文写明代码放置路径 | Agent 自建目录，位置不一致 |
| 5 | **外部依赖已列出** | `## 前置准备` 中写明服务/token/文件 | AFK 运行时缺依赖 |
| 6 | **mock/E2E 策略已声明** | AC 明确测试方式 | Agent 冲突（mock vs 真连） |
| 7 | **SDK 用法可参考** | 已在 prompt 可用依赖表格中记录 | Agent curl 探索 API |
| 8 | **验收不含主观判断** | "质量好""结构完整"等须量化 | 不可验证 |
| 9 | **跨 issue 接口引用** | 引用其他 issue 产出时确认 blocked_by。修改共享类型时：所有 list/dict 字段须为 Optional 带默认值，禁止破坏性变更。修改被多模块引用的文件后，必须在 master 上跑一次全量测试确认 0 失败 | 接口不稳定，共享类型变更导致下游模块测试失败 |
| 10 | **架构约束已引用** | 参考项目架构文档中的不可变规则 | 实现违反架构设计 |
| 11 | **AC 覆盖集成层** | 不只是模块清单，有关联模块的编配层 | Agent 只做零件不做组装 |
| 12 | **Scope 边界已声明** | `## Scope` 章节写清 In（哪些文件/目录）和 Out（不碰哪些） | Agent 探索无关代码 |
| 13 | **运行环境已声明** | frontmatter 含 `needs_llm` / `needs_vision` / `needs_pdf` / `needs_docker` / `needs_mcp`，明确 issue 需连接哪些外部服务 | 纯函数 issue 不部署 LLM，浪费 token |
| 14 | **测试范围已指定** | frontmatter 含 `test_files`，指向精确的测试文件路径，禁止默认跑全量 | 误判已有失败用例，虚耗重试 |

## 后置检查（提交 ready 前）

| # | 检查项 | 方法 |
|---|--------|------|
| 1 | type 与 AC 是否矛盾 | grep type: + 读 AC 清单，确认 AFK 不含主观验收 |
| 2 | 所有 AC 可验证 | 逐条确认：测试能测或人能观测 |
| 3 | 前置准备章节存在且完整 | 确认包含外部服务/文件/token/验证方式 |
| 4 | blocked_by 不循环 | 跟踪依赖链，确认引用的 issue 在合理状态 |
| 5 | secret.json 中有对应 token | 运行检查命令 |
| 6 | Git 仓库已同步 | git pull 无冲突 |
| 7 | 目录不与已有模块冲突 | ls 对应路径 |
| 8 | 不引用不存在/未完成的接口 | 检查被引 issue 状态 |
| 9 | 跨 issue Rubric 引用匹配 | 确认引用适用 |
| 10 | mock/E2E 策略与 AC 不矛盾 | 同一 issue 内测试策略一致 |
| 11 | needs_* 与 AC 一致 | 确认 needs_llm/vision/pdf 与 AC 清单匹配 |
| 12 | test_files 存在且可访问 | 确认测试文件路径有效 |
| 13 | **变更共享类型需跑全量回归** | 凡涉及被多模块引用的文件，在 master 合入后跑一次全量 `pytest` 确认 0 失败 |
| 14 | **共享字段使用 Optional** | 所有共享类型中可能被多个下游独立赋值的字段，必须使用 `Optional` + 默认值，禁止必填字段扩张 |

## 速查卡

```
___ Issue 质量宪法 14 项 ___
 1  ≤1d        2  type正确
 3  AC可测      4  目录指定
 5  外部依赖     6  mock/E2E
 7  SDK参考     8  无主观
 9  接口稳定     10 架构约束
11  集成覆盖     12 Scope边界
13  needs声明    14 test_files
```

## 引用方式

写 issue 时在正文开头引用：

```markdown
## 质量自检

- [ ] 工时 ≤1d
- [ ] type: AFK（无人工步骤）
- [ ] AC 全可量化
- [ ] 目录已指定
- [ ] 前置准备已列
- [ ] mock/E2E 已声明
- [ ] SDK 参考已记录
- [ ] 验收无主观
- [ ] blocked_by 已设
- [ ] 架构约束已引用
- [ ] 集成层已覆盖
- [ ] Scope 边界已声明
- [ ] needs_* 已声明
- [ ] test_files 已指定
- [ ] 共享类型变更后全量回归
- [ ] 共享字段使用 Optional
```
